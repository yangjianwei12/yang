Detailed Description

Handover Profile.

Implementation of the protocol between earbuds during handover.

Functionality of the acceptor device. The acceptor is the device that accepts the link(s) being handed over by the initiator of the handover. In a standard handover with role switch, the acceptor is the secondary role earbud, it accepts the links handed over by the primary role earbud.

Handover Profile private declarations


The functionality to perform handover as a primary.

Handover Profile device declaration.

Handover Profile appsP1 interface.

Core procedures used by the handover profile to perform a handover.

◆ HandoverProfile_HandleSubsystemVersionInfo()

void HandoverProfile_HandleSubsystemVersionInfo ( const MessageSubsystemVersionInfo *  info)

#include <adk/src/domains/bt/profiles/handover_profile/handover_profile.h>

Handle subsystem version information.

Parameters
[in]infoThe subsystem version information.
Note
Handover profile will only handover if the firmware and patch version info on the primary and secondary earbuds matches. Matching firmware is required since handover involves a binary state transfer from primary to secondary. A firmware mismatch means the secondary may incorrectly interpret the binary state data leading to invalid state.

◆ HandoverProfile_Handover()

handover_profile_status_t HandoverProfile_Handover ( void  )

#include <adk/src/domains/bt/profiles/handover_profile/handover_profile.h>

Performs handover to the peer device.

This is a blocking call and returns after handover is complete or if any failure. handover_profile_status_t for return types.

Handover profile will determine which handsets are connected and attempt to handover all connected handsets.

Returns
handover_profile_status_t. Returns,
  1. HANDOVER_PROFILE_STATUS_SUCCESS if handover is successful.
  2. HANDOVER_PROFILE_STATUS_SUCCESS if peer earbud is not conected. Caller need to invoke HandoverProfile_PeerConnect and try again.
  3. HANDOVER_PROFILE_STATUS_HANDOVER_VETOED if any of the clients vetoed handover.
  4. HANDOVER_PROFILE_STATUS_HANDOVER_TIMEOUT if handover terminated due to timeout.
  5. HANDOVER_PROFILE_STATUS_HANDOVER_FAILURE if any other failure occured.

Callback registration for handover events in acceptor role.

This API facilitates the registration of callback functions designed to manage Handover events within the acceptor (Secondary) role. Also this API allows only one client to register callback functions on acceptor role. Subsequent call to this API shall overwrite the previous registration.

The same API shall be used to De-register the callbacks by passing a NULL parameter

◆ HandoverProfile_TerminateSdpPrimitive()

void HandoverProfile_TerminateSdpPrimitive ( void  )

#include <adk/src/domains/bt/profiles/handover_profile/handover_profile.h>

Cancel any queued SDP search sent by this module.

If handover profile has sent an SDP search request and is waiting for the response, this function will immediately cancel the SDP search request, or flush it from the queue if it is still queued in the connection library.

The intention is that this function is called directly before the peer ACL is force disconnected. Otherwise, a queued SDP search request could trigger the ACL to be re-connected in order to do the SDP search.
