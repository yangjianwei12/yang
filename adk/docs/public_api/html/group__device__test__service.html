Implementation

The normal flow of control is shown below

Authentication

The service uses authentication to avoid unexpected access. A python example is provided to show the authentication method. device_test_service_auth_example.py

Use of device database

The device test service cleans up after itself if possible. The details of the devices used for driving the device test service should be removed from the device database.

This is done by adding a property, device_property_device_test_service, to an entry in the device database when we pair with the device.

Implementing new commands

Any new commands should be added to the file device_test_parse.parse. This defines the command and any paramaters and also defines the name of the function to be called.

When the service is running and a matching command is detected - the function is called with a Task and a structure containing any parameters. All responses to the command should use the Task supplied so that the response can be linked to the command. This may be important for some interfaces to the device test service.

If a command should supply any informaion as part of a response that should be sent using DeviceTestService_CommandResponse() or DeviceTestService_CommandResponsePartial().

Handling of a command should be completed using a call to DeviceTestService_CommandResponseOk() or DeviceTestService_CommandResponseError(). The function DeviceTestService_CommandResponseOkOrError() is provided for use if the result of a command can either be TRUE or FALSE.

DTS has 2 sub-channels which are multiplexed over the DTS case comms channel. The sub-channels are 'management' and 'tunnel' and are identified by the case comms message ID (MID) used in DTS case comms channel messages.

The management sub-channel is used exclusively by the case to communicate with DTS on the Earbuds.

The tunnel sub-channel is used by external hosts, to tunnel standard DTS messaging through the case to the Earbuds. This is analogous to standard DTS messaging direct to the Earbuds over other supported DTS transports such as SPP.

This illustrates that commands are allowed to take some time to process and may use tasks. When the response is finally generated, the current state of the device test service should be checked, using DeviceTestService_CommandsAllowed().

Note
This file contains more comments than normal. The functions are also extracted in the documentation as an example.
        For peripherals the commands can monitor the sensor for a time, allowing
        for user interaction. Battery and temperature are read and returned
        immediately.

        The type of sensor is abstracted behind the interface. The existence of
        a sensor is controlled by build defines, so the software here is 
        conditionalised accordingly. If a particular sensor type is not supported
        then the commands will return ERROR.

The Device Test Service secret key is a key that must be known to users wishing to authenticate for access. The secret key is held in device_test_service_auth_key which is compiled into the application.

Authentication follows a challenge response sequence using AES-CMAC. An example python script is provided that could be used at the host side. device_test_service_auth_example.py

◆ DeviceTestService_CommandResponse()

void DeviceTestService_CommandResponse ( Task  task,
const char *  response,
unsigned  length 
)

#include <adk/src/services/device_test/device_test_service.h>

Send a string response to the last command

The task passed should match that sent in the function call.

Responses will be terminated with \r\n automatically.

The response supplied should be a normal C string, with a zero termination. The supplied length is used so it would be legal to send a string with no termination if the length is correct. 

'O''K', length 2
"OKAY", length 2 will just use "OK"
Note
The maximum response length is limited. Consider the limit to be about 100 characters (
See also
#DEVICE_TEST_SERVICE_MAX_RESPONSE_LEN). If a response is longer than this then DeviceTestService_CommandResponsePartial should be used for the response.
Note
The string sent in the response will be the smallest of Its actual length as a null-terminated string The length supplied as a parameter #DEVICE_TEST_SERVICE_MAX_RESPONSE_LEN
Interim responses such as this should eventually be followed by an OK (or ERROR)
See also
DeviceTestService_CommandResponseOk
Parameters
taskThe task information sent for the incoming command
responseThe response to be sent.
lengthThe length of the response. 0 is allowed, in which case the length of the string is used.

Send part of a single response to the last command

The task passed should match that sent in the function call.

Most responses should be less than 100 characters,

See also
#DEVICE_TEST_SERVICE_MAX_RESPONSE_LEN for the exact limit). Responses within that limit should use DeviceTestService_CommandResponse() which adds necessary characters at the beginning and end of the message.

If a response is longer than the limit then this function can be used to separate the response into multiple parts. The first part of the response should have first_part set TRUE, and the final part should have last_part set TRUE.

Note
The string sent in the response will be the smallest of Its actual length as a null-terminated string The length supplied as a parameter #DEVICE_TEST_SERVICE_MAX_RESPONSE_LEN
This function can be called with both first_part and last_part set TRUE. This is equivalent to DeviceTestService_CommandResponse().
See also
DeviceTestService_CommandResponse() for other details of parameters.
